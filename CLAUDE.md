## Project status

This repository is in the **design phase**. As of now it contains only documentation
(`README.md`, `docs/analysis.md`) and no source code — there is no `Cargo.toml`, `src/`,
or build tooling yet. The first implementation work will be scaffolding the Rust project.

`docs/project/high-level-analysis.md` is the authoritative design document; read it before
implementing, along with `docs/project/low-level-analysis.md` for the proposed implementation
blueprint. The summary below captures the key decisions.

## What this project is

`uptime-kuma-rs` is a lightweight Rust service that periodically polls an
[Uptime Kuma](https://github.com/louislam/uptime-kuma) instance and re-exposes its monitoring
data via a clean, stable read-only REST API. It exists because Uptime Kuma only exposes its
data through an undocumented, version-unstable socket.io interface; this service replaces that
with something external consumers can depend on.

## Architecture (planned)

Core pattern: **poll → cache → serve**.

- **Stack:** Rust + Axum (HTTP, on Tokio async runtime). Chosen for a small footprint
  (target < 10MB resident) and a mature async ecosystem.
- **Data source — public status-page JSON, not socket.io:** Uptime Kuma has no general read
  REST API and its socket.io interface is unofficial/unstable. The primary source is the public
  status-page endpoints (`GET /api/status-page/:slug` and `/api/status-page/heartbeat/:slug`):
  no auth, clean JSON, but they require a published status page and only expose 24h uptime.
  `/metrics` (Prometheus) and socket.io are documented fallbacks. Incidents are derived locally.
- **Storage:** in-memory snapshot (hot read path, the default) + **SQLite** for durable heartbeat
  history, needed to compute 7d/30d uptime and reconstruct incidents (the status-page API only
  gives 24h). **Redis is optional**, off by default, used only for multi-replica deployments
  (shared snapshot cache behind one poller). Single instance needs only in-memory + SQLite.
- **Partial uptime windows:** on a fresh DB, 7d/30d are computed from whatever history exists and
  returned alongside a per-window `coverage` field so consumers can distinguish a complete figure
  from a partial one (rather than returning `null` or a misleading bare number).
- **Deployment:** single container + a volume for the SQLite file; SQLite is a file, not a
  service. `docker-compose.yml` runs app+SQLite by default; Redis is behind a compose profile.
- **Configurable poll interval:** TTL set via config file or environment variable, trading data
  freshness against load on the upstream instance.
- **Read-only / non-destructive:** The service never writes to the Uptime Kuma instance.

## Planned REST API

All endpoints are `GET` and read-only:

- `GET /api/monitors` — current status and latency of each monitor
- `GET /api/uptime` — uptime % over standard time windows (24h / 7d / 30d)
- `GET /api/incidents` — history of incidents (monitors that went down)

See `docs/analysis.md` §4 for response schemas.

## Security model

Designed to run on a **private network** (homelab, VPN, LAN), not exposed directly to the
internet. Uptime Kuma credentials are never re-exposed in responses. CORS is configurable to
restrict consumers, and optional API-key auth via the `X-Api-Key` header protects endpoints in
semi-exposed environments.

## Downstream integration

This service is the Uptime Kuma data source for
[`homelab-api`](https://github.com/1brecane/homelab-api), which combines it with Proxmox data
behind a single `/api/dashboard` endpoint. Keep `uptime-kuma-rs` generic and reusable —
portfolio-specific concerns belong in `homelab-api`, not here.

## Commands

No build tooling exists yet. Once the project is scaffolded with `cargo init`, the standard
Rust workflow applies:

- Build: `cargo build` (release: `cargo build --release`)
- Run: `cargo run`
- Test all: `cargo test`
- Run a single test: `cargo test <test_name>`
- Lint: `cargo clippy --all-targets`
- Format: `cargo fmt`

## Documentation convention

Keep documentation well-structured under `docs/`, never in the repo root. Use these
subdirectories by purpose:

- `docs/project/` — core project docs (e.g. `docs/project/high-level-analysis.md`, the
  authoritative design doc, and `docs/project/low-level-analysis.md`)
- `docs/superpowers/` — docs generated by skills (plans, specs, brainstorming output, etc.)
- `docs/memory/` — docs that serve as durable memory / reference notes

Place each new doc in the directory that matches its purpose.

## Configuration

Secrets and local config go in `.env` (gitignored). Do not commit it.
